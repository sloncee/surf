# Этап 3 — Тестирование: «Гончарная мастерская "Глина"»

Проверка трёх реализованных фич (`02-development/features/`) на соответствие
требованиям аналитики: сгенерированы тест-кейсы, по ним найдены и исправлены
баги.

## Структура

| Папка/файл | Содержимое |
|---|---|
| `test-cases/test-cases.md` | 20 тест-кейсов по US-001..US-004, формат Given/When/Then |
| `prompts/prompts.md` | Промпты, использованные при генерации тест-кейсов и поиске багов |

Сами баги задокументированы не здесь, а в `02-development/bugs/` — по той же
причине, по которой фичи лежат в `02-development/features/`: исправление
бага — это правка кода приложения, а не аналитический артефакт. Здесь —
только тест-кейсы, которые их обнаружили.

## Как читать

1. `test-cases/test-cases.md` — таблицы тест-кейсов по каждой фиче со
   статусом (✅ / ❌ / —) и ссылкой на требование.
2. Для кейсов со статусом ❌ — переход по ссылке на баг в
   `02-development/bugs/BUG-00x-....md` (симптом, причина, исправление,
   промпты, чек-лист ручной проверки).

## Результат

- Тест-кейсов: 20.
- Багов найдено и исправлено: **2**.
  - **BUG-001** — порог отмены (`cancellation_deadline_at`) считался от
    момента брони, а не от начала занятия (нарушение FR-D3 и допущения №1
    из `customer-questions.md`). Существенно сокращал клиенту окно на
    бесплатную самостоятельную отмену.
  - **BUG-002** — ссылка «позвонить в мастерскую» (FR-D4) отдавала в `href`
    нецифровые символы (`tel:+7(812)555-12-34`), что не соответствует
    формату `tel:`-URI и рискует не сработать как ссылка для звонка на части
    устройств.
- Оба фикса проверены вручную и не сломали соседние сценарии (US-002,
  US-003, US-004 в целом) — детали в чек-листах внутри карточек багов.

## Как перепроверить

```bash
cd 02-development/app
npm install
npm run dev
```

Чек-листы ручной проверки — в `02-development/bugs/BUG-00x-....md`
(раздел «Ручная проверка»). Для BUG-001 дополнительно есть отдельный
скрипт, демонстрирующий формулу до/после фикса (см. промпты в
`prompts/prompts.md`).
