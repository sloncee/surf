# Фича: US-002 — Запись на мастер-класс

## Симптом / цель

Клиент видит список ближайших занятий (US-001) и должен иметь возможность
записаться на выбранное занятие сам, без звонка/переписки с мастерской, с
выбором «свои инструменты / прокат» и корректной обработкой ситуации, когда
места или прокатный фонд закончились.

## Требования

- User story: **US-002** (`01-analysis/2-requirements/user-stories.md`) —
  и US-001, т.к. без списка слотов записываться не на что.
- Функциональные требования: **FR-C1–FR-C5**
  (`01-analysis/2-requirements/functional-requirements.md`), также
  FR-B1/FR-B4/FR-B5 (список слотов).
- Экран: SCR-002/SCR-003/SCR-004 из `5-design-brief/design-brief.md` —
  упрощено до одного экрана, см. «Реализация».
- Эндпоинты контракта: `GET /slots`, `POST /bookings`
  (`01-analysis/4-api/api-contract.md`).
- Бизнес-правило: атомарность проверки мест — на бэкенде (R-004); клиент
  только корректно обрабатывает отказ (409/422).

## Реализация

Файлы:
- `app/src/screens/SlotsListScreen.tsx` — список слотов + встроенная форма
  записи (выбор «свои / прокат», кнопка «Записаться»).
- `app/src/api/client.ts` — методы `getSlots()`, `createBooking()`.
- `app/src/api/mockAdapter.ts` — обработчики `GET /slots` и `POST /bookings`,
  включая проверку `free_seats`/`free_rental_kits` и коды 409/422.
- `app/src/api/fixtures/slots.json` — тестовые слоты, в т.ч. один без мест
  (`slot-2`) и один с ограниченным прокатом (`slot-1`), чтобы можно было
  вручную воспроизвести обе ошибки.

Подход: без отдельных экранов «Карточка слота»/«Оформление записи» — форма
записи прямо в карточке списка. Это самый короткий путь к рабочей фиче;
разнести по экранам можно позже, когда фич станет больше и один экран станет
слишком перегруженным.

Что замокано: `POST /bookings` в `mockAdapter.ts` — уменьшает `free_seats`/
`free_rental_kits` слота в памяти, создаёт `Booking` со статусом `confirmed`
и `cancellation_deadline_at = +2 часа` (MVP-допущение из
`customer-questions.md` №1). Бронь хранится только в памяти процесса
(переживает сессию вкладки, не диск) — этого достаточно для демонстрации.

## Промпты

> Реализуй фичу US-002 — Запись на мастер-класс полностью, выбрав самый короткий путь 
> к работающему результату. Убери всё лишнее: никаких избыточных экранов, абстракций 
> или заделов «на будущее». В итоге фича должна быть готова целиком — от API до UI и обработки ошибок.
> Это шаблон для двух следующих фич, поэтому код должен быть простым, читаемым и типизированным 
> — чтобы его легко было копировать и адаптировать. Отдельно опиши, как запустить проект 
> и проверить работу: какие команды выполнить, что должно открыться, какие сценарии пройти.

## Ручная проверка

- [ ] `npm run dev` в `app/` → открыт список из 3 предстоящих занятий
      (прошедший слот `slot-past` в список не попадает — FR-B1).
- [ ] Слот с `free_seats = 6` (третий в списке) → нажать «Записаться» с
      «Свои» → появляется «Вы записаны», счётчик свободных мест уменьшается
      на 1; запись видна во «Мои записи» → «Предстоящие» (FR-C5).
- [ ] Тот же слот → выбрать «Прокат» → «Записаться» → успех,
      `free_rental_kits` уменьшается на 1.
- [ ] Слот с `free_seats = 0` (`slot-2`) → кнопка «Записаться» неактивна
      (задизейблена) — FR-C1. Это превентивная защита в UI; серверная
      проверка на 409 (race condition, R-004) в мок-адаптере тоже
      реализована, но её физически не воспроизвести в одиночку в браузере —
      достаточно свериться с кодом `mockAdapter.ts` (ветка `free_seats <= 0`).
- [ ] Слот с местами, но без проката (`slot-3`, `free_rental_kits = 0`) →
      выбрать «Прокат» (недоступен) или попытаться записаться без своих
      инструментов → ошибка «Свободного проката не осталось…» — FR-C3.
      С «Свои» тот же слот бронируется успешно.
- [ ] Обновить страницу браузера → все брони «теряются» (мок хранит их в
      памяти процесса) — ожидаемое поведение для этого шага, не баг.
