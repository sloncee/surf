# Решение по стеку и подходу

## Контекст

`3-design/architecture-plan.md` фиксирует стек только индикативно (Flutter/React
Native — кроссплатформенная мобильная разработка) и явно разрешает скорректировать
выбор на этапе разработки без изменения аналитики (раздел 5).

Ограничения этого конкретного этапа:
- разработка ведётся с использованием бесплатных версий Claude и Cursor;
- бэкенда по факту нет — нужен мок-слой;
- цель этапа — минимум 3 рабочие фичи + тесты, а не production-ready мобильное
  приложение.

## Решение

**Веб-приложение: React + TypeScript + Vite**, вместо Flutter/React Native.

Причины:
- запускается командой `npm install && npm run dev`, без Xcode/Android
  Studio/эмуляторов — это быстрее всего поднять и проверить в Cursor;
- Cursor лучше всего работает именно с TS/JS-экосистемой;
- "мобильность" на этом этапе выполняется через mobile-first вёрстку
  (см. `design-brief.md`, принцип 1), а не через нативный рантайм — для целей
  учебного MVP и демонстрации 3 фич этого достаточно;
- при необходимости перейти на React Native слой `api/` и `types/` переносится
  почти без изменений (это чистый TS, без зависимости от DOM).

## Мок вместо реального бэкенда

- Единая точка входа — `app/src/api/client.ts`, экраны вызывают только её.
- `app/src/api/mockAdapter.ts` эмулирует пути и коды ответов из
  `4-api/api-contract.md` (включая 409/422/401 из таблиц обработки ошибок) на
  основе фикстур из `app/src/api/fixtures/*.json`.
- Искусственная задержка (~300 мс) в моке — чтобы сразу проектировать состояния
  загрузки, а не добавлять их последним штрихом.

## Что осознанно не делаем сейчас

- Роутер (react-router) — переключение экранов через простой `useState` в
  `App.tsx` (сейчас: «Расписание» / «Мои записи»), чтобы не тянуть лишнюю
  зависимость.
- State-менеджер (Redux/Zustand) — пока состояние локальное на экран.
- Реальная авторизация/токены — `AuthScreen` пока заглушка.
