# Уточняющие вопросы к заказчику

Бриф намеренно неполный/противоречивый. Ниже — вопросы, разделённые на **закрытые**
(ответ получен в виде «Уточнения по скоупу» в брифе) и **открытые** (по ним в MVP
принято рабочее допущение — Assumption, — которое стоит подтвердить с заказчиком до
или во время разработки).

## Закрытые вопросы (ответ есть в брифе)

| # | Вопрос | Ответ |
|---|---|---|
| 1 | Что именно нужно реализовать — всё приложение целиком (включая бэкенд, кабинет мастера, админку) или только часть? | Только клиентское мобильное приложение + API для него. Бэкенд, расписание, кабинеты мастера/владельца уже существуют (R-004, R-028) |
| 2 | Кто гарантирует, что не будет двойных броней? | Существующий бэкенд, атомарной проверкой при бронировании; клиент лишь обрабатывает отказ (R-004) |
| 3 | На сколько дней вперёд по умолчанию показывать расписание? | 7 дней, с фильтром на больший период; при пустом расписании — empty state (R-027) |
| 4 | Что происходит с бронью клиента, если мастерская отменяет занятие целиком? | Бронь не удаляется, получает статус «Отменено мастерской» с причиной + push; повторная запись на этот слот запрещена (R-008) |
| 5 | Есть ли историческая база клиентов/записей для миграции? | Нет, проект учебный, миграция вне скоупа (R-015) |

## Открытые вопросы (по MVP принято допущение, требует подтверждения)

| # | Вопрос | Почему важно | Рабочее допущение (Assumption) для MVP |
|---|---|---|---|
| 1 | Какой именно порог «поздней» отмены клиентом (в брифе только пример «за 10 минут — уже проблема»)? Что происходит при попытке отменить позже порога — запрет, предупреждение, санкция? | Влияет на FR отмены и на UX (нельзя просто «злить клиентов», но нужно защищать мастерскую от простоя) | Свободная отмена — не позднее чем за **2 часа** до начала. Позже — кнопка отмены недоступна, показывается пояснение и контакт мастерской для форс-мажорных случаев. Порог задаётся бэкендом (приходит в контракте слота), приложение не хардкодит число |
| 2 | Как определяется «постоянный клиент» и что именно ему полагается (бейдж, приоритет, скидка)? | Влияет на модель профиля и на UI | В MVP клиентское приложение только отображает флаг/статус `loyalty_status`, если он присутствует в ответе API профиля; логика начисления — на бэкенде, вне скоупа |
| 3 | Какие каналы уведомлений нужны на старте — только push, или ещё SMS/e-mail? | Влияет на архитектуру уведомлений | MVP: только push-уведомления (напоминание о записи + уведомление об отмене мастерской). SMS/e-mail — бэклог |
| 4 | Когда именно клиенту предлагается оценить мастера — сразу после окончания занятия (по времени), только по явному действию из «Моих записей», или и то, и другое? | Влияет на use case и на нотификации | И то, и другое: push-напоминание «оцените мастера» после окончания слота + возможность оценить вручную из карточки прошедшей записи в любой момент (одна оценка на бронь) |
| 5 | Может ли клиент оценить занятие, если он не пришёл (no-show) или отменил бронь? | Нужно для валидации use case «Оценка» | Нет: оценка доступна только для брони со статусом «Завершено» (клиент присутствовал) |
| 6 | Нужна ли авторизация/регистрация, и по какому идентификатору (телефон/e-mail)? В брифе не упомянуто явно | Без этого нельзя спроектировать API/экран входа | MVP: регистрация/вход по номеру телефона + одноразовый код (OTP) — минимальный порог входа для мобильного приложения массового пользователя |
| 7 | Можно ли отменить бронь не только клиенту, но кто-то из мастерской делает это за него по телефону — нужно ли клиенту видеть эту отмену как отдельный статус? | Пограничный случай на стыке с админкой | Такая отмена выполняется в существующей инфраструктуре и придёт клиенту как обычная отмена мастерской (статус «Отменено мастерской», R-008) — отдельный статус не нужен |
| 8 | Что показывать, если у клиента нет ни одной активной/прошедшей записи? | Empty state для экрана «Мои записи» | Понятный empty state с CTA «Найти мастер-класс» |
| 9 | «Гончарных кругов всего 10» — это лимит вместимости одного слота на круге, или общий физический фонд мастерской, который делится между несколькими параллельно идущими занятиями на круге (если расписание такое допускает)? | Влияет на корректность понимания домена; для самого клиентского приложения не блокирующий вопрос, так как `free_seats` в любом случае приходит от бэкенда готовым числом (R-004), но важен для правильной постановки задачи бэкенду/админке | Для клиентского приложения принимается нейтральное допущение: приложение просто отображает `total_seats`/`free_seats`, которые вернул API, не делая собственных предположений о том, как эти 10 кругов распределены между слотами |

> Вопросы 3 и 6 являются архитектурно значимыми (влияют на состав API и экранов) —
> рекомендуется подтвердить их с заказчиком в первую очередь.
