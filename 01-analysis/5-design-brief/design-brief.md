# Дизайн-требования (design brief)

## Принципы

1. **Mobile-first, одна рука.** Ключевые действия — запись и отмена — доступны без
   долгого скролла и лишних экранов (NFR-05).
2. **Честные пустые и ошибочные состояния.** Не «белый экран», а понятный текст +
   призыв к действию (пустое расписание, отмена мастерской, конфликт мест).
3. **Прозрачность статусов брони.** Клиент всегда должен однозначно понимать
   текущий статус своей записи (цвет + текст, а не только цвет).
4. **Не давить на клиента при отмене.** Порог отмены объясняется, а не просто
   блокируется молча (см. FR-D4).

## Экраны (для клиентского приложения, роль — только Клиент)

| ID | Экран | Ключевое содержимое |
|---|---|---|
| SCR-001 | Вход/регистрация (OTP) | Ввод телефона → ввод кода |
| SCR-002 | Список слотов | Фильтры (программа, мастер, даты), карточки слотов на 7 дней, empty state |
| SCR-003 | Карточка слота | Программа, мастер (с рейтингом), время, цена, свободные места/прокат, кнопка «Записаться» |
| SCR-004 | Оформление записи | Выбор «свои инструменты / прокат», подтверждение |
| SCR-005 | Мои записи | Вкладки: предстоящие / прошедшие / отменённые, empty state. *(Уточнение: бронь со статусом «Неявка» (`no_show`) отображается во вкладке «прошедшие» с отдельной пометкой, отличной от «Завершено» — без формы оценки, FR-E3.)* |
| SCR-006 | Карточка брони | Статус, детали слота, кнопка отмены (или пояснение, если недоступна), форма/результат оценки |
| SCR-007 | Профиль | Имя, телефон, статус лояльности (если есть), настройки push |

## Ключевые UX-сценарии, требующие отдельной проработки состояний

- **BS-001.** Успешная запись — экран/тост подтверждения.
- **BS-002.** Конфликт при бронировании (места закончились) — некритичная ошибка с
  предложением альтернативы, а не тупиковый экран.
- **BS-003.** Подтверждение отмены брони — модальное окно с явным указанием, что
  действие необратимо.
- **BS-004.** Уведомление об отмене мастерской — акцентный статус в «Моих записях»
  + возможность посмотреть причину.

> Детальная проработка макетов (Figma) — отдельный этап, не входящий в текущую
> аналитическую поставку; здесь зафиксированы только требования к содержимому
> и поведению экранов, достаточные для перехода к разработке MVP.
