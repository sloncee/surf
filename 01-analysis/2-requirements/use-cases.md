# Use Cases

## UC-001. Запись клиента на мастер-класс

**Актор:** Клиент
**Предусловия:** клиент авторизован; существует хотя бы один слот со свободными местами.
**Постусловия (успех):** создана бронь в статусе «Подтверждена»; `free_seats` слота
уменьшено на 1 (и `free_rental_kits`, если выбран прокат).

**Основной сценарий:**
1. Клиент открывает список слотов на ближайшие 7 дней.
2. Клиент выбирает слот и открывает карточку с деталями.
3. Клиент нажимает «Записаться».
4. Клиент выбирает: свои инструменты / прокат.
5. Клиент подтверждает запись.
6. Приложение отправляет запрос на создание брони в API.
7. Бэкенд атомарно проверяет наличие свободных мест/фонда и создаёт бронь.
8. Приложение показывает подтверждение; бронь появляется в «Мои записи».

**Альтернативные сценарии:**
- **4a.** Прокатного фонда не осталось → опция «прокат» неактивна, доступно только
  «свои инструменты» (или блокировка записи, если программа требует инструменты
  мастерской, а своих у клиента нет — сообщение с пояснением).
- **7a.** Места закончились между шагом 2 и шагом 7 (гонка запросов) → бэкенд отклоняет
  запрос → приложение показывает сообщение «Места уже заняты» и предлагает вернуться
  к списку слотов (см. FR-C4, R-004).
- **7b.** Сетевая ошибка при отправке запроса → приложение показывает повтор попытки,
  бронь не считается созданной, пока не получено подтверждение от API.

## UC-002. Отмена брони клиентом

**Актор:** Клиент
**Предусловия:** у клиента есть бронь в статусе «Подтверждена» с датой начала в будущем.
**Постусловия (успех):** бронь переходит в статус «Отменено клиентом»; место (и
прокатный фонд, если был занят) возвращается в свободный пул слота.

**Основной сценарий:**
1. Клиент открывает «Мои записи» → карточку предстоящей брони.
2. Клиент нажимает «Отменить запись».
3. Приложение проверяет `cancellation_deadline_at` (порог, пришедший из API).
4. Если порог не пройден — приложение показывает подтверждающий диалог.
5. Клиент подтверждает отмену.
6. Приложение отправляет запрос на отмену в API; бэкенд обновляет статус и
   освобождает место.
7. Приложение отображает бронь в статусе «Отменено клиентом».

**Альтернативные сценарии:**
- **3a.** Порог отмены пройден → кнопка «Отменить» неактивна, показано пояснение
  («Отмена возможна не позднее чем за N часов до начала») и контакт мастерской.
- **6a.** Слот уже был отменён мастерской раньше, чем пришёл запрос клиента (гонка
  состояний) → приложение обновляет статус до актуального («Отменено мастерской»)
  и показывает соответствующее сообщение, а не ошибку.

## UC-003. Отмена занятия мастерской (форс-мажор) — сторона клиента

**Актор:** система бэкенда (инициирует администратор/мастерская — вне скоупа
клиентского приложения), Клиент — получатель уведомления.
**Предусловия:** существует слот с одной или несколькими подтверждёнными бронями;
слот отменён средствами существующей инфраструктуры (вне скоупа, R-008).
**Постусловия:** все связанные брони получают статус «Отменено мастерской» с причиной;
затронутые клиенты получают push; повторная запись на слот невозможна.

**Основной сценарий (со стороны клиентского приложения):**
1. Бэкенд помечает слот как отменённый и обновляет статусы связанных броней (вне
   скоупа приложения).
2. Приложение получает push-уведомление «Занятие отменено» с причиной.
3. Клиент открывает уведомление или заходит в «Мои записи».
4. Приложение показывает бронь в статусе «Отменено мастерской» с указанием причины.
5. Если клиент пытается записаться на этот слот повторно — приложение не позволяет
   выбрать его (слот скрыт из доступных для записи / помечен как отменённый).

**Альтернативные сценарии:**
- **2a.** Push не доставлен (нет разрешения/нет сети) → при следующем открытии
  «Моих записей» клиент всё равно видит актуальный статус (NFR-09).

## UC-004. Оценка мастера после занятия

**Актор:** Клиент
**Предусловия:** у клиента есть бронь в статусе «Завершено», оценка по ней ещё не
поставлена.
**Постусловия (успех):** создана оценка мастера, привязанная к брони; агрегированный
рейтинг мастера пересчитан (на бэкенде).

**Основной сценарий:**
1. Клиент получает push-напоминание оценить мастера (или заходит в «Мои записи»
   самостоятельно).
2. Клиент открывает карточку завершённой брони.
3. Приложение показывает форму оценки (1–5 звёзд + необязательный комментарий).
4. Клиент выставляет оценку и отправляет.
5. Приложение отправляет запрос в API; бэкенд сохраняет оценку.
6. Карточка брони обновляется — показана поставленная оценка вместо формы.

**Альтернативные сценарии:**
- **2a.** Бронь в статусе «Отменено клиентом/мастерской» или ещё не началась → форма
  оценки не отображается вовсе (см. FR-E3).
- **2b.** Оценка для этой брони уже существует → вместо формы показывается уже
  выставленная оценка (без возможности повторной отправки, FR-E2).

## UC-005. Просмотр расписания с фильтрами и пустым результатом

**Актор:** Клиент
**Предусловия:** клиент авторизован.
**Постусловия:** клиент видит релевантный (отфильтрованный) список слотов либо
корректный empty state.

**Основной сценарий:**
1. Клиент открывает главный экран со списком слотов на 7 дней вперёд (по умолчанию).
2. Клиент применяет фильтр (программа, мастер и/или диапазон дат).
3. Приложение запрашивает у API отфильтрованный список.
4. Приложение показывает результат.

**Альтернативные сценарии:**
- **4a.** Результат пуст → показан empty state «Пока нет доступных занятий» с
  предложением изменить фильтр/период (R-027).
