# Нефункциональные требования (НФТ / NFR)

| ID | Категория | Требование |
|---|---|---|
| NFR-01 | Производительность | Список слотов и «Мои записи» должны загружаться не дольше 2 секунд на стабильном мобильном соединении (4G) |
| NFR-02 | Надёжность данных | Приложение не хранит и не рассчитывает доступность мест самостоятельно — всегда доверяет актуальному ответу API (single source of truth — бэкенд, R-004), чтобы избежать рассинхронизации |
| NFR-03 | Устойчивость к конфликтам | При отказе бэкенда из-за отсутствия мест (гонка запросов) приложение обрабатывает ошибку 409/аналог понятным для пользователя сообщением, без падения экрана |
| NFR-04 | Доступность сети | Мастерская — помещение с потенциально нестабильным Wi-Fi/LTE; экран «Мои записи» кэширует последние загруженные данные локально и показывает их в режиме offline с пометкой «данные могли устареть» |
| NFR-05 | Юзабилити | Интерфейс мобильный (iOS/Android), рассчитан на использование одной рукой; ключевые действия (запись, отмена) — не более 2–3 нажатий от списка слотов |
| NFR-06 | Локализация | MVP — только русский язык интерфейса |
| NFR-07 | Безопасность | Авторизация по OTP; все запросы к API — по HTTPS с токеном сессии (JWT/аналог); персональные данные клиента передаются только по защищённому каналу |
| NFR-08 | Приватность данных | Обработка персональных данных клиентов (ФИО, телефон) соответствует требованиям применимого местного законодательства о персональных данных — согласие на обработку запрашивается при регистрации. *(Исправлено: ранее требование ссылалось конкретно на 152-ФЗ РФ, хотя страна/юрисдикция мастерской в брифе не указана — это было неподтверждённым допущением, а не фактом из брифа. Конкретный нормативный акт нужно подставить после уточнения юрисдикции у заказчика.)* |
| NFR-09 | Доставка push-уведомлений | Push-уведомления — best effort (через FCM/APNs), не являются единственным каналом критичной информации: статус отменённой мастерской брони в любом случае виден в «Моих записях» |
| NFR-10 | Совместимость | Поддержка актуальных версий iOS и Android (2 последние мажорные версии на момент релиза) |
| NFR-11 | Наблюдаемость | Ключевые ошибки клиентского приложения (сбой бронирования, сбой отмены) логируются на клиенте для последующей диагностики (без сбора лишних персональных данных) |
| NFR-12 | Масштабируемость | Приложение проектируется с расчётом на рост числа мастерских/филиалов в будущем, хотя MVP — одна мастерская (не должно быть жёстко зашитых предположений о единственной локации в клиентском коде) |
| NFR-13 | Поддерживаемость | API-контракт — единственная граница между приложением и бэкендом; изменения в бэкенде не должны требовать пересборки приложения, если контракт не меняется |
