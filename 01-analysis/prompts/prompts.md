# Промпты, использованные на этапе «Аналитика»

## Контекст задания от организаторов летней школы

> Задание направлено на изучении использования ИИ в промышленной разработки. Оно
> разделено на три этапа — аналитика, разработка и тестирование. В рамках брифа для
> гончарной мастерской необходимо пройти все три этапа. Основные задачи в рамках
> брифа: 
> 1) сгенерировать с ИИ требования к MVP (user stories + основные сценарии);
> 2) получить от ИИ архитектурный план и схему данных; 
> 3) реализовать хотя бы 3 фичи; 
> 4) сгенерировать тест-кейсы, найти и исправить от 1 до 3 багов. 
> 
> Для каждой задачи и бага завести .md документ, кратко описать симптом/цель, найти требования,
> реализовать/доработать, прикрепить все отправленные промпты, проверить вручную,
> сделать commit.

## Промпт для этапа «Аналитика»

> Первый этап - аналитика. На основе брифа необходимо описать домен, сформулировать
> требования (ФТ и НФТ), создать User Story и Use Case, учесть критерии приемки,
> составить архитектурный план и схему данных, описать требования к api и дизайну
>
> Результат должен быть представлен в виде структурированного набора .md файлов.
> Информация должна быть полной и непротиворечивой, при этом не избыточной.
> Структура шагов может быть скорректирована, главное условие - выполнение основных
> пунктов задания
>
> Можно опираться на пример из файла (аналогичный этап для проекта с другой
> тематикой), при этом необязательно жестко следовать его образцу

**Входные данные:** `brief-pottery.md` (бриф заказчика, включая позднейшее
уточнение по скоупу — R-004, R-008, R-015, R-027, R-028) + референс структуры
из `surfstudio-summer-school-2026-*.txt` (аналогичный этап для проекта другой
тематики, изучен для формата, но не скопирован дословно).

## Как был обработан промпт

1. Изучена структура файла-примера (дерево каталогов `01-analysis/...`) как
   ориентир по составу артефактов.
2. Бриф прочитан целиком, включая блок «Уточнение по скоупу», зафиксированы все
   явные бизнес-правила и границы скоупа (R-004, R-008, R-015, R-027, R-028).
3. Выделены неоднозначности брифа (точный порог поздней отмены, механика
   «постоянного клиента», наличие авторизации и т. п.) — оформлены как открытые
   вопросы с рабочими допущениями (`1-elicitation/customer-questions.md`), чтобы
   не блокировать дальнейшую работу, но сохранить прозрачность допущений.
4. Сформирован набор `.md` файлов по разделам: домен → требования (BR/FR/NFR) →
   User Story/Use Case → архитектура/данные → API → дизайн-требования, без
   дублирования одной и той же информации между файлами (например, бизнес-правила
   заведены один раз в `domain-description.md` и переиспользуются по ID в
   остальных документах).

## Промпт для ревью этапа «Аналитика»

> Проведи ревью проделанной на первом этапе работы. Найди грубые неточности и
> исправь их. Отправь измененные файлы и добавь список файлов, не требующих
> изменений.

**Как был обработан промпт:** выполнена построчная сверка каждого файла с брифом
(grep + чтение) и перекрёстная сверка файлов друг с другом по ключевым терминам
(статусы брони, числовые ограничения, ссылки на нормативные акты). Найденные
несостыковки и внесённые правки зафиксированы отдельно в `../requirements-review.md`
вместе со списком файлов, не потребовавших изменений.
