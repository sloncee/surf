# Архитектурный план

## 1. Границы системы

```mermaid
flowchart LR
    subgraph OUT["Вне скоупа (существует)"]
        BE[Существующий бэкенд:\nрасписание, мастера, программы,\nатомарное бронирование]
        AdminUI[Кабинет владельца/администратора]
        MasterUI[Кабинет мастера]
        BE --- AdminUI
        BE --- MasterUI
    end

    subgraph SCOPE["Текущая поставка"]
        API[API-контракт для клиента\n(фасад над бэкендом)]
        App[Клиентское мобильное приложение]
    end

    Push[(Push-сервис\nFCM / APNs)]

    App -- HTTPS/REST+JSON --> API
    API -- проксирует/использует данные --> BE
    API -- инициирует отправку --> Push
    Push -- push-уведомление --> App
```

Ключевой архитектурный принцип: клиентское приложение **не содержит бизнес-логики
проверки доступности мест** — эта ответственность принадлежит бэкенду (R-004).
Приложение — тонкий клиент над API-контрактом.

## 2. Компоненты клиентского приложения

| Слой | Ответственность |
|---|---|
| **UI / Presentation** | Экраны (список слотов, карточка слота, бронирование, «Мои записи», карточка брони, оценка, профиль) |
| **State management** | Хранение состояния экранов, кэш последнего успешного ответа API для offline-показа (NFR-04) |
| **API client** | HTTP-клиент к API-контракту, обработка ошибок (409 — конфликт мест, 401 — не авторизован и т.д.), retry для сетевых сбоев |
| **Auth module** | OTP-регистрация/вход, хранение и обновление токена сессии |
| **Push module** | Регистрация push-токена устройства на бэкенде, обработка входящих push (напоминание, отмена мастерской, предложение оценить) |
| **Local cache** | Легковесное локальное хранилище (например, ключ-значение/встроенная БД) для offline-режима «Моих записей» |

## 3. Взаимодействие с бэкендом

- Единственная точка интеграции — **API-контракт** (см. `4-api/api-contract.md`).
- Формат — REST/JSON поверх HTTPS, авторизация — Bearer-токен (JWT или аналог),
  полученный после OTP-входа.
- Все проверки бизнес-инвариантов (вместимость, прокатный фонд, двойные брони,
  пороги отмены) выполняются на бэкенде; приложение отображает то, что вернул API,
  и корректно обрабатывает коды ошибок.
- Обновления статусов, инициированные не самим клиентом (например, отмена
  мастерской), доставляются двумя путями: push-уведомление (best effort, NFR-09) и
  обычный запрос состояния при следующем открытии экрана «Мои записи» — так
  гарантируется, что клиент в любом случае увидит актуальный статус.

## 4. Нефункциональные соображения в архитектуре

- **Offline-устойчивость** (NFR-04): кэш последнего списка «Моих записей», отметка
  «данные могли устареть», если запрос к API не удался.
- **Безопасность** (NFR-07): токен сессии хранится в защищённом хранилище устройства
  (Keychain/Keystore), не в открытом виде.
- **Расширяемость** (NFR-12): в модели данных клиента предусмотрено поле
  идентификатора мастерской/локации, даже если сейчас она одна, — чтобы не зашивать
  предположение о единственной мастерской в клиентский код.
- **Разделение ответственности с бэкендом** (NFR-13): любые изменения бизнес-правил
  (например, порог отмены) параметризуются через API-контракт (поля вроде
  `cancellation_deadline_at`), а не хардкодятся в приложении.

## 5. Технологический стек (индикативно, не императивно)

Поскольку в брифе платформа не зафиксирована, а требование — «в основном с телефона»,
предполагается кроссплатформенная разработка (Flutter/React Native) с единой
кодовой базой iOS/Android; выбор конкретного стека — вне зоны аналитики и может
быть скорректирован на этапе разработки без изменения данного документа.
