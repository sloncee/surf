# API-контракт для клиентского приложения

> Формат — REST/JSON поверх HTTPS. Авторизация — Bearer-токен, полученный после
> OTP-входа. Этот документ описывает **требования к контракту** (какие данные и
> операции нужны клиенту); конкретная OpenAPI-спецификация может быть развёрнута
> из этой таблицы на этапе разработки.

## Auth

| Метод | Путь | Описание |
|---|---|---|
| POST | `/auth/otp/request` | Запросить одноразовый код на номер телефона |
| POST | `/auth/otp/verify` | Подтвердить код, получить access/refresh токен |
| POST | `/auth/refresh` | Обновить access-токен по refresh-токену |

## Профиль

| Метод | Путь | Описание |
|---|---|---|
| GET | `/profile` | Получить профиль клиента (включая `loyalty_status`, если есть) |
| PATCH | `/profile` | Обновить имя/e-mail |
| POST | `/profile/push-token` | Зарегистрировать push-токен устройства |

## Программы и мастера (только чтение)

| Метод | Путь | Описание |
|---|---|---|
| GET | `/programs` | Список программ (новички/круг) с описанием |
| GET | `/masters/{id}` | Профиль мастера с агрегированным рейтингом |

## Расписание

| Метод | Путь | Описание |
|---|---|---|
| GET | `/slots?date_from&date_to&program_id&master_id&page` | Список слотов. По умолчанию `date_from=сейчас`, `date_to=+7 дней` (R-027) |
| GET | `/slots/{id}` | Детали слота (полная карточка) |

**Ответ `/slots`:** пустой массив `items` при отсутствии слотов на период → на
клиенте рендерится empty state (FR-B4), это не ошибка, а штатный ответ 200.

## Брони

| Метод | Путь | Описание |
|---|---|---|
| POST | `/bookings` | Создать бронь: `slot_id`, `own_tools` (bool) |
| GET | `/bookings?status=upcoming\|past\|cancelled` | Список броней клиента (группы для UI-вкладок, см. маппинг ниже) |
| GET | `/bookings/{id}` | Детали брони, включая `cancellation_deadline_at` |
| POST | `/bookings/{id}/cancel` | Отменить бронь |
| POST | `/bookings/{id}/rating` | Оставить оценку по завершённой брони: `score`, `comment` |

**Маппинг UI-группы `status` → реальный статус брони.** *(Исправлено: ранее группы
`upcoming/past/cancelled` не были явно сопоставлены с полным набором статусов
брони из модели данных, что оставляло неясным, куда относятся `completed` и
`no_show`.)*

| Группа в query-параметре | Реальные статусы брони (из `BOOKING.status`) |
|---|---|
| `upcoming` | `confirmed` (дата начала в будущем) |
| `past` | `completed`, `no_show` |
| `cancelled` | `cancelled_by_client`, `cancelled_by_studio` |

Право на оценку (`POST /bookings/{id}/rating`) есть только у брони со статусом
`completed`; для `no_show` бэкенд должен возвращать 422/403 (FR-E3).

### Обработка ошибок бронирования (важно, R-004)

| Код | Ситуация | Ожидаемое поведение клиента |
|---|---|---|
| 200/201 | Успех | Показать подтверждение |
| 409 Conflict | Мест/прокатного фонда не осталось на момент подтверждения | Показать сообщение «Места уже заняты», предложить другой слот (FR-C4) |
| 422 | Невалидные данные запроса (например, попытка выбрать прокат при `free_rental_kits = 0`) | Показать причину, вернуть к форме |
| 401 | Токен истёк/невалиден | Тихий refresh или повторный вход |

### Обработка отмены (FR-D3, FR-D4)

| Код | Ситуация | Ожидаемое поведение клиента |
|---|---|---|
| 200 | Успешная отмена | Обновить статус на «Отменено клиентом» |
| 409/422 | Порог отмены (`cancellation_deadline_at`) уже прошёл | Кнопка отмены недоступна ещё до запроса (клиент сверяет с `cancellation_deadline_at` локально), запрос — как дополнительная защита |
| 409 | Слот уже отменён мастерской | Обновить статус до актуального «Отменено мастерской», без показа ошибки |

## Уведомления (push, инициируются бэкендом)

Форматы payload push, которые должно уметь обрабатывать приложение:

| Тип | Когда отправляется | Данные в payload |
|---|---|---|
| `booking_reminder` | За N часов до начала (конфигурация на бэкенде) | `booking_id`, `slot_id`, `start_at` |
| `booking_cancelled_by_studio` | Сразу после отмены слота мастерской | `booking_id`, `cancellation_reason` |
| `rate_master_prompt` | После завершения занятия | `booking_id`, `master_id` |

## Общие требования к контракту

- Все списковые эндпоинты — с пагинацией (`page`, `page_size`, `meta.total`).
- Все денежные суммы — целые числа в минимальных единицах валюты (например, копейки)
  либо целые рубли — фиксируется на этапе разработки.
- Все даты/время — ISO 8601 с таймзоной.
- Контракт — единственная граница ответственности с бэкендом (R-004, R-015):
  приложение не обращается к бэкенду напрямую, минуя этот API.
